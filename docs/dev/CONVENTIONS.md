# CONVENTIONS.md — стандарты кода, пре-комментариев и логирования

Обязательны для всякого кода в `src/`. Проверяются линтером
(`python scripts/lint_precomments.py`) и reviewer-ом перед коммитом.

## 1. Шапка модуля (docstring файла)

Каждый .py-файл в `src/ariadna/` начинается с docstring:
назначение (2–3 строки); вход → выход в терминах `contracts.py`; зависимости;
инварианты («никогда не пишет в граф напрямую, только через graph/loader»);
ссылка на паспорт `docs/dev/modules/<модуль>.md`.

## 2. Пре-комментарии (основной стандарт документирования функций)

Спецификация функции пишется КОММЕНТАРИЕМ ПЕРЕД сигнатурой (не docstring-ом):
блок, встреченный до тела, работает как правило, через которое читается реализация;
фиксированные ключи делают блок машинно-проверяемым.

**Полная форма — для публичных функций** (имя без `_`):

```python
# ─── generate_chunks ──────────────────────────────────────────
# Назначение: режет нормализованный текст на чанки с перекрытием,
#   сохраняя границы предложений (колонтитулы уже удалены в ingest).
# Входные связи: contracts.DocumentText; config.chunking (size, overlap)
# Выходные данные: list[contracts.Chunk] — id, doc_id, text, span, lang
# Уровень: ✅ реализовано (A-04, worklogs/ingest.md#2026-07-03)
def generate_chunks(doc: DocumentText, cfg: ChunkingConfig) -> list[Chunk]:
```

Обязательные ключи полной формы: `Назначение:`, `Входные связи:`, `Выходные данные:`,
`Уровень:`.

**Короткая форма — для приватных хелперов** (имя с `_`): одна строка `Назначение:`
+ строка `Уровень:`.

**Словарь уровней:** `📋 нужно реализовать · ✅ реализовано · 🔒 запрет на редактирование`.

**Дисциплина:**
- Пре-комментарий обновляется ТЕМ ЖЕ изменением, что и функция. Расхождение блока
  с кодом — инцидент (устаревшей спецификации агент поверит охотнее, чем коду).
- Смена уровня 📋→✅ — обязанность агента, завершившего задачу; сверяется со STATUS.md.
- Уровень 🔒 ставит только оркестратор (Fable/solver): весь `contracts.py` и принятый
  критический код. Diff, трогающий 🔒-функцию без санкции оркестратора, отклоняется.
- Каркас 📋-функций (сигнатура + пре-комментарий + `raise NotImplementedError`) пишет
  оркестратор при постановке задачи — агент получает контракт до реализации.

## 3. Прочие правила кода

- **Инлайн-комментарии** — только ограничения, которые код не выражает («Яндекс-PDF
  из Журналов содержат колонтитулы — режем до чанкинга, иначе мусор в эмбеддингах»).
  Комментарии «что делает следующая строка» запрещены.
- **Модуль ≤ ~350 строк** — файл целиком помещается в контекст агента вместе
  с паспортом и контрактами. Больше — сигнал к декомпозиции.
- Магические значения — только именованными константами с комментарием-обоснованием.
- Типизация обязательна на публичных функциях; структуры между модулями — только
  модели из `contracts.py`.

## 4. Логирование (восстанавливает место и суть поломки)

- Формат: JSON Lines, `logs/pipeline/<run_id>.jsonl`; стандартный `logging`
  + JSON-formatter, без экзотических зависимостей.
- Обязательные поля события: `ts, run_id, module, stage, doc_id (если применимо),
  level, event, detail`. Сквозные `run_id` + `doc_id` — трасса одного документа
  через весь конвейер одним grep.
- Коды ошибок: `INGEST-xxx / EXTRACT-xxx / GRAPH-xxx / SEARCH-xxx / UI-xxx /
  DEPLOY-xxx`, реестр — `docs/dev/ERRORS.md`. Новый путь ошибки = новый код
  в реестре тем же пакетом изменений.
- Правило «ошибка = контекст»: событие ERROR обязано включать данные для
  воспроизведения (имя файла, chunk_id, промпт-хеш, ответ LLM усечённый до 500 симв.).

## 5. Тесты

- Тесты модуля — в `tests/<модуль>/`, пишет module-tester (не автор кода).
- Быстрые и офлайновые: LLM/Neo4j мокируются; интеграционные — отдельно, по задаче.
